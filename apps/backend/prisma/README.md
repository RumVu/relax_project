# Prisma Workspace

This project keeps Prisma schema source files split by domain in `prisma/schema-parts/`.

Prisma CLI configuration lives in `prisma.config.ts`.

## Source of truth

- Edit `prisma/schema-parts/*.prisma.part`
- Do not edit `prisma/schema.prisma` by hand
- `prisma/schema.prisma` is generated by `prisma/scripts/build-schema.cjs`

## Common commands

- `npm run prisma:sync-schema`
- `npm run prisma:validate`
- `npm run prisma:generate`
- `npm run prisma:db:push`
- `npm run prisma:migrate:dev -- --name init`
- `npm run prisma:migrate:deploy`
- `npm run prisma:seed`
- `npm run prisma:studio`

`prisma:validate` and `prisma:generate` use a safe local fallback `DATABASE_URL` if one is not set yet.
Database commands that actually connect to a DB still expect a real `DATABASE_URL`.

## Migration history

Canonical migration history is currently squashed into one migration:

- `20260519000000_initial_schema`

This migration creates the current live `public` schema from an empty database.
The local development database migration history has been normalized to this
single migration, so fresh and existing local databases use the same Prisma
history shape.

Do not re-add the old split migrations on top of this branch without a deliberate
migration-history reset. Mixing the old two-step history with the squash makes
Prisma report pending/duplicate migrations and can trigger reset prompts.

## Legacy database merge

The current Prisma-owned schema is `public` in `digital_cigarette_break`.
The old database snapshot is no longer kept as a live schema. It was exported
to a local, gitignored backup before being dropped from the active database.
Only `public` should remain after running migrations.

If a future audit needs the old snapshot, restore the backup into a temporary
database or recreate the `legacy` schema, then run:

```bash
DB_URL="${DATABASE_URL%%\?*}"
psql "$DB_URL" -f prisma/scripts/merge-legacy-public.sql
npm run prisma:seed
```

The merge script copies supported legacy data into the current schema:

- users, profiles, preferences, default companions
- mood check-ins, journals, relax sessions, weekly stat inputs
- cozy quotes, achievements, badges
- subscription tiers, tier features, tier limits
- social/gamification contracts: friends, challenges, points, streaks, levels
- operational contracts: rate limits, cache, search, events, admin logs
- insight/content contracts: analytics snapshots, AI insights, recommendations,
  content ratings, meditation guides/sessions

Keep `public` as the only live source of truth for the app runtime. Do not keep
`legacy` mounted in normal local or deployed databases.

See `../../../docs/10-operational-readiness.md` for the canonical ownership map
between mood stats, streak ledgers, events, relax sessions, meditation history,
weather, storage, and optional provider readiness.

## Suggested workflow

1. Edit one or more files in `prisma/schema-parts/`
2. Run `npm run prisma:validate`
3. Run `npm run prisma:generate`
4. Create a migration with `npm run prisma:migrate:dev -- --name <migration_name>`
